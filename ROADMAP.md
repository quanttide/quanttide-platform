# ROADMAP

## 当前阶段：文档体系建立（v0.3.x）

三层文档链路（BRD→PRD→ADD）和 MyST 文档站基建已跑通，两个写作技能在握。

### 已完成

- 逆向文档化方法论：代码 → ADD → PRD → BRD
- BRD/PRD 写作技能化（`product-brd`、`product-prd`）
- MyST 文档站模板 + GitHub Pages CI/CD 工作流
- 子仓库文档分工范式：README（陈述性）/ CONTRIBUTING（程序性）/ AGENTS（元认知）

### 范式未完成

- **IXD 层无标准**：交互设计文档的叙事规范未提炼为技能，各子仓库写法不一
- **QA 层无标准**：质量保证文档的验证规范未提炼为技能，无法闭环验证 BRD 的问题是否解决
- **链路缺闭环**：BRD→PRD→ADD→QA 的正向链路最后一环缺失，文档体系停留在设计阶段，无法回答"问题解决了吗"
- **迁移无指南**：旧格式到新格式的转换缺乏 checklist 和模板，子仓库自行摸索成本高

---

## 方向一：技能补齐

### 目标

补齐文档链路末端的两个技能槽位，使五层体系（BRD→PRD→IXD→ADD→QA）具备完整的标准定义。

### 产出

- `product-ixd` SKILL：交互设计文档的叙事规范
- `product-qa` SKILL：质量保证文档的验证规范，验收标准可追溯到 PRD 的 Given-When-Then
- 链路追溯矩阵模板：每层之间的引用关系显式化

### 检验标准

- 新增模块能按 skills 索引找到对应写作规范，无需口头上问"这个文档怎么写"
- 新技能发布后，有子仓库自愿试点（非主仓库驱动）

---

## 方向二：迁移框架

### 目标

不替子仓库做迁移，但提供迁移的范式支撑——产出旧格式→新格式的转换指南和模板。

### 产出

- 迁移 checklist：旧文档 → 新格式的自检清单
- 模板仓库：一个最小可用的新格式 docset 模板，子仓库可 fork 或参考

### 检验标准

- 子仓库参照 checklist 独立完成迁移，无需主仓库介入
- 迁移前后信息损耗可评估

---

## 方向三：主仓库文档归位

### 背景

主仓库 `docs/` 目前有 BRD（空）、PRD（产品边界草稿）、ADD（跨模块架构标准），是在旧体系下的遗留结构。新体系下子仓库已有独立文档站，主仓库文档需要重新定义职责。

### 目标

主仓库文档从"存放领域内容"演化为"存放范式定义"——不重复子仓库的产品细节，只回答跨子模块的问题。

### 产出

| 当前 | 演化后 |
|------|--------|
| `docs/index.md` | 仓库层级关系（保留） |
| `docs/brd/`（空） | 补充：为什么需要产品矩阵 |
| `docs/prd/`（产品边界草稿） | 重写为产品组合策略：功能归哪个子仓库、组合逻辑 |
| `docs/add/`（跨模块架构） | 保留，清理冗余 |

### 核心原则

- 子仓库 `docs/` 回答"这个产品怎么做"
- 主仓库 `docs/` 回答"为什么这些产品要放在一起"

### 注意

不与方向一/二抢优先级。主仓库文档归位依赖技能补齐和迁移框架就绪——只有子仓库的写作标准稳定了，主仓库的归位才有参照系。

---

## 不变原则

- 主仓库不持有任何子仓库的具体工作项——不在 ROADMAP 里写"重写 XX 模块"，那是子仓库自己的事
- 主仓库只提供范式支撑（skill、模板、checklist），具体执行下放到子仓库
- 范式是否有效，看子仓库是否自愿采用，不靠主仓库推动
