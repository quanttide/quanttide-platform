# 领域事件传递机制

## 定位

领域事件是跨平台协作的核心机制，对应 BRD 场景四"跨领域业务流程的平台断裂"的架构方案。

核心原则：**每个域只发布事件，不关心谁消费；只订阅需要的事件，不过问谁生产。** 域之间通过事件契约耦合，不产生运行时依赖。

## 事件格式

采用 [CloudEvents](https://cloudevents.io/) 规范统一事件格式：

| 字段 | 说明 | 示例 |
|------|------|------|
| `source` | 事件来源域 | `qtcloud.sales` |
| `type` | 事件类型 | `com.qtcloud.sales.deal.closed` |
| `subject` | 事件主体 ID | `deal-123` |
| `data` | 业务负载 | `{ dealId, amount, customerId }` |
| `time` | 发生时间 | ISO 8601 |

## 事件契约

每个域用 AsyncAPI 定义自己发布和订阅的事件。契约是域间的唯一耦合点，变更需遵循 Consumer-Driven Contract——提供方不得单方面变更事件格式破坏消费者。

## 可靠发布

采用 Outbox 模式：

1. 业务操作与事件写入本地数据库（同一事务）
2. 独立进程读取未发送事件并发布到总线
3. 发布成功后标记为已发送

确保事件不丢失、不重复，与业务事务一致。

## 跨域流程模式

**Choreography（默认）**：无中心协调器，每个域自行决定行为。

```
销售云 → deal.closed 事件
商务云 → 收到，创建合同
资产云 → 收到，准备交付
```

**Saga（高一致性场景）**：每个域执行本地事务，失败时触发补偿。适合涉及多域数据变更的场景。

## 方案推荐

| 方案 | 适用场景 |
|------|---------|
| 事件总线（Redis Stream / NATS） | 轻量级跨服务传递，多语言友好，初期首选 |
| 消息队列（RabbitMQ） | 复杂路由和投递保障 |
| 事件流平台（Kafka） | 高吞吐、事件溯源、离线消费，规模增长后迁移 |

初期从事件总线起步，事件格式标准先行，中间件后续按需替换。
