# ADD 工作方式

## 核心原则

- **ADD 应为代码库结构，而非理论模型**：目录名直接对应代码目录（如 `apps/` 映射主仓库 `apps/`），避免使用"领域层/接口层"等抽象概念
- **目录结构本身就是架构声明**：目录传达的信息比内容更多。有实体就建目录，但预留目录也是有效的架构声明——它们定义了将来内容的边界。
- **ADD 是事后记录，非预先设计**：先有代码和产品，再记录架构决策。`index.md` 描述架构风格（给所有人看），`README.md` 做目录索引（给维护者看）
- **语义不重叠**：同一件事只在一个目录下出现。如云服务属于基础设施，就归入 `infra/`，不另建同级

## 典型错误

- 按理论分层建目录（如 `domains/interfaces/`）→ 与代码实际结构脱节
- 概念作为目录名而非目录结构（如 `cloud_services/` 与 `infra/` 同级）→ 区分成本高
