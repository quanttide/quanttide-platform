# 微服务

使用容器部署平台部署。

## 划分

大体遵循一个子应用对应一个微服务的原则，允许子应用有多个微服务。比如，量潮课堂有主站和管理后台两个子应用，分别有对应的主要微服务，同时主站还有代码执行器微服务支持。

每个微服务使用一个子域名。计划购买`quanttideapi.com`并制定域名分配规则。
原定`api.quanttide.com`或者`api.<service>.quanttide.com`计划取消。

## 编排和治理

### 跨域流程编排

默认采用 Choreography（舞蹈编排）：每个域根据自身订阅的事件自主决定行为，无中心协调器。

```
销售云 → 发布 deal.closed 事件
商务云 → 收到事件，创建合同
资产云 → 收到事件，准备交付
```

仅在跨域数据一致性要求高的场景使用 Saga 模式——每个域执行本地事务，失败时触发补偿操作。

### API 网关

网关处理跨域横切关注点：认证鉴权、限流、协议转换、请求路由。域服务只暴露领域 API，不处理跨域逻辑。

### 事件治理

以 AsyncAPI 契约作为跨域协作的唯一规范接口。域间不共享数据库、不直连内部 API、不共享领域模型。契约变更遵循 Consumer-Driven Contract 模式。
