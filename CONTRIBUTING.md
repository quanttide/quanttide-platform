# 量潮产品设计语言

本文档总结量潮各子产品（qtcloud-asset、qtcloud-health 等）共有的设计风格和取向，作为新产品设计的参考基线。

## 核心理念

让意图可追溯，让变更可控制，让系统可解释。

## 设计原则

七条原则分为四组：契约（系统与外部的约定）、架构（系统如何组织）、底线（不可妥协的安全）、进化（人与系统如何共生）。

### 契约

### 文档即契约

把文档当作产品的外部接口，而非事后记录。先有契约，再有实现。

- qtcloud-asset：契约即接口，YAML 经 Pydantic Schema 转为验证引擎，契约加载即验证，失败立即报错
- qtcloud-health：BRD、PRD、IXD、ADD、QA 五层文档即产品契约，每层回答不同问题，QA 证明交付承诺已履行
- 共同取向：文档不是说明文，是约束

### 架构

### 分层解耦

同样的分层哲学，在不同领域反复出现。职责边界清晰，每层只知道自己该知道的事。

- qtcloud-asset：CLI 四层解耦，入口层到契约层到工作流层到操作层，层间通过类型通信，严禁跨层调用
- qtcloud-health：系统四层解耦，前端层到接口层到业务层到数据层，层间通过接口通信
- 共同风格：每层有明确的职责和禁区，通过接口而非实现细节通信

### 领域归位

每个功能属于且仅属于一个领域产品，不属于管理外壳（qtadmin）或任何聚合仓库。聚合仓库只负责编排路由和组合视图，**不持有领域逻辑**。

- qtadmin 是管理后台，职责是跨产品的管理界面编排，不是领域功能的收容所
- qtcloud-asset 是资产领域产品，拥有资产契约、资产注册表等完整领域能力
- 共同风格：发现功能放错位置时及时迁移，而非在聚合层堆积领域代码

### 唯一事实源

对数据一致性的执着。一份数据只存一处，其他位置引用它。

- qtcloud-asset：测试数据集中在 assets/fixtures，通过 conftest.py fixture 引用，严禁复制样例数据
- qtcloud-health：PRD 定义核心数据模型和 JSON Schema，让前后端对齐，数据模型是系统的骨架
- 共同取向：消除该用哪份数据的歧义

### 可解释可追溯可验证

对黑盒的警惕贯穿所有项目。任何决定都要能回答为什么。

- qtcloud-asset：QA 即交付证明，结构是准则到判定到证据，不记录测试过程，只回答证明了什么
- qtcloud-health：AI 输出必须包含置信度、推理过程和辩驳提示，每条认知扭曲识别都有推理过程
- 共同风格：拒绝不可解释的行为，每个输出都有据可查

### 底线

### 安全护栏优先于功能

安全不是附加项，是架构核心。失败快速暴露，绝不吞错误。

- qtcloud-asset：契约加载即验证，失败立即报错，绝不静默降级
- qtcloud-health：三道防线，危机关键词检测、AI 解析管道、数据加密存储，安全机制贯穿各层
- 共同取向：安全防护不是附加功能，而是架构的核心支柱

### 进化

### AI 辅助不替代

对人类控制权的尊重。AI 是助手，不是决策者。

- qtcloud-asset：AI 转换管道使用 Ollama 模型将日志转换为结构化蓝图，但人工审查和确认是最终环节
- qtcloud-health：明确写着 AI 只提供解析和建议，用户始终拥有最终控制权
- 共同取向：AI 提供输入，人类做决策

### 文档是活文档

对写完就扔的拒绝。文档随产品进化，不追求一次完美，追求持续对齐。

- qtcloud-asset：V2.0 规划明确演进路径，架构变更记录设计理由
- qtcloud-health：CONTRIBUTING 规定所有文档应该是活文档，每季度审查，欢迎团队提出改进建议
- 共同取向：文档与产品同步进化

## 项目约定

### 版本与发布

- **SemVer**: `v{major}.{minor}.{patch}`
- **Phase**: Exploration (0.0.x) → Validation (0.x.y) → Release (x.y.z)
- **Changelog**: Keep a Changelog 格式
- **Commits**: Conventional Commits
- **Pre-release**: `v0.x.y` 阶段，patch 用于修正/小调整，minor 用于新能力体系（方法论建立、规范落地、可复用的模板）

## 产品设计检查清单

设计新产品时，自检以下维度。

- 契约：产品的外部接口是什么，有明确的契约定义且契约可验证
- 分层：系统分几层每层职责是什么，每层有明确的职责和禁区，无跨层调用
- 归位：功能属于哪个领域产品，领域逻辑是否放在聚合仓库中，聚合仓库不持有领域代码
- 安全：安全防护在哪里失败如何暴露，安全机制贯穿各层，失败快速暴露
- 可解释：系统的决定能解释吗，每个重要输出都有据可查
- 事实源：数据存在哪里有重复吗，每份数据只存一处其他位置引用
- AI 角色：AI 做什么人做什么，AI 提供建议人类做决策
- 文档：文档如何维护多久审查一次，有 CONTRIBUTING 定期审查机制
